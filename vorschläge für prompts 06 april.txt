The approach with 2 dicts seems sound but lets move them both to one meta-dict. 

A user would initialize a xet.grid2 to use basic functionality and can be sure that the data is saved in the patcher, just like it is right now. the xet.grid2 ui object can exist with internal data only and without a dict yet. the data consists of:
- the information which row is of which kind (a row can be of kind step, which is stepgrid button, mom, which is momentary button, and toggle)
- the cell values for all three kinds (step, mom, toggle) for the whole grid, that means we will have 3 sets of grid value data
- the row colors for the full step rows
- row and column colors if a row is a mom or toggle row (we can also set the color of each cell indivudally (by sending messages into xet.grid2)).

generally, we will have a layout menu that can be opened by clicking a button, which displays two drop down menu overlayed over each row - the first drop down menu selects the row type, the second drop down menu selects the displayed row data. here, if we have not saved a grid by the "savegrid (gridname)" command yet, the second drop down menu is grayed out, not selectable. that is because we are only accessing the internal data of the xet.grid2 object here. if we select "step" as a row type, the ui grid behaves like it does now. if we select "mom" or "toggle", the seqstart and seqlength numbers left and right of the grid are not visible for this row, and changing pages does not have any influence for this row. 

If a user wants to be able to use the "savegrid (gridname)" functionality and save the different gridnames & their data in the patcher, they would need to also initialize a [dict xet.grids @embed 1]. The "grids" are representing banks, so to say. This single meta storage dict has 2 keys on the top level: "Data::", where all the data is stored and "Layout::", where all the mappings (all the grids) are stored. The whole notion of banks gets collapsed into the term "grid". a grid is a list of row mappings, deciding which row displays which row data. 

when we click "savegrid" we enter the name of the grid that we want to store the current layout to. the previously unnamed (because fresh) rows get a unique identifier. if, for example, we then click savegrid a second time we can either save to another new grid with a different gridname, or overwrite the grid we have already saved (if the gridname is the same). like this, we can build layouts and save them as "grids" into the Layout:: part of [dict xet.grids @embed 1]. once we have saved at least one grid into the dict xet.grids, the second right drop down menu in the layout menu becomes selectable/visible. now we can select rows by their unique identifiers to be displayed in the current ui grid (and then save that, of course, with "savegrid")

when we call savegrid, a new list of row mappings with unique identifiers is generated, but we can still select which row should point to the first grid and override the row identifiers. If we want to save these mappings, we just call savegrid but without an argument.

savegrid does 2 things:
1. create an entry gridname in Layout:: and 2. write the current data into Data::
